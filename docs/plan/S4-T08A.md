# Task: S4-T08A Python YC API bootstrap tool

## Goal
Create a Python utility that can be called from GitHub Actions to ensure API Gateway exists, apply/update its OpenAPI spec, and return stable outputs consumed by downstream CI steps (`NOTIFY_ENDPOINT`, gateway identifiers).

> Scope of this task is **planning only**. Implementation in `tools/yc_bootstrap.py` will be done in a separate task/PR.

## Why this change
Current deploy workflow depends on `yc` CLI setup and shell parsing. This increases CI fragility due to:
- installation/network variability;
- implicit state in CLI profiles;
- non-typed JSON parsing in shell.

A Python API client flow should provide deterministic auth handling, structured logging, and explicit outputs.

## Proposed utility contract

### CLI interface (draft)
```bash
python tools/yc_bootstrap.py \
  --folder-id "$YC_FOLDER_ID" \
  --gateway-name "$YC_API_GW_NAME" \
  --function-id "$YC_FUNCTION_ID" \
  --service-account-id "$YC_SERVICE_ACCOUNT_ID" \
  --spec-template infra/apigw.yaml \
  --public-base-url "$NOTIFY_PUBLIC_BASE_URL" \
  [--iam-token "$YC_IAM_TOKEN" | --sa-key-file /path/key.json] \
  [--dry-run] \
  [--output-env]
```

### Outputs (stdout)
Utility should print machine-readable env-style lines:
- `YC_API_GW_ID=<id>`
- `YC_API_GW_NAME=<name>`
- `NOTIFY_ENDPOINT=<https://.../notify>`
- `YC_API_GW_DOMAIN=<domain>` (optional, for diagnostics)

When `--output-env` is specified, only key-value output should be emitted to simplify parsing in Actions.

## Required YC API operations
1. **Authentication**
   - If IAM token provided: use it directly.
   - Else if SA key provided: call IAM token endpoint to exchange key for short-lived IAM token.
2. **Gateway discovery by name**
   - Query API Gateway list in folder and locate by exact `gateway_name`.
3. **Gateway create (if missing)**
   - Create gateway with rendered OpenAPI spec.
4. **Gateway update (if exists)**
   - Update existing gateway spec in-place.
5. **Deployment readiness check**
   - Poll operation status for create/update until done or timeout.
6. **Endpoint resolution**
   - Resolve canonical domain and produce `NOTIFY_ENDPOINT`.

## Inputs and configuration
Mandatory:
- auth: `YC_IAM_TOKEN` **or** `YC_SA_JSON_CREDENTIALS` / `--sa-key-file`;
- `YC_FOLDER_ID`;
- `YC_API_GW_NAME`;
- `YC_FUNCTION_ID`;
- `YC_SERVICE_ACCOUNT_ID`;
- spec template path (`infra/apigw.yaml`).

Optional:
- `NOTIFY_PUBLIC_BASE_URL` override for externally fronted domain;
- operation timeout and poll interval;
- `--dry-run` (no mutating calls).

## Rendering rules for OpenAPI template
Before create/update:
- replace placeholders for function id and service account id;
- replace optional public base URL if provided;
- fail fast if unresolved placeholders remain (`<YOUR_...>` pattern).

## Logging and safety
- Never print raw SA key or full IAM token.
- Mask sensitive values in diagnostics.
- Emit clear error categories: auth, permissions, missing resource, invalid spec, operation timeout.
- Return non-zero exit code on any failed required operation.

## Rollback strategy
If deploy fails after gateway mutation:
1. Re-run tool with last known good spec snapshot (`--spec-template <previous>`).
2. If snapshot unavailable, revert workflow to previous commit that used CLI path.
3. Validate endpoint via smoke test and only then continue deploy pipeline.

Operational guardrails:
- Keep current CLI-based path available during transition (parallel fallback).
- First runs in CI should default to `--dry-run` until permissions are verified.
