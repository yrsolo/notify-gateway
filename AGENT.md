# AGENT.md — рекомендации агенту по разработке

Этот файл задаёт рабочий протокол для агентской разработки в репозитории `notify-gateway`.

## 1) Базовый цикл работы (обязательно)

1. Посмотреть, на какой стадии сейчас находится проект:
   - `docs/execution/STAGE_PLAN.md`
   - `docs/execution/CURRENT_STAGE_TASKS.md`
   - `docs/execution/PROGRESS.md`
2. Выбрать **текущую активную стадию** и проверить её Definition of Done (DoD).
3. Разбить текущую стадию на атомарные задачи (если ещё не разбита):
   - каждая задача должна быть проверяемой за 1 PR;
   - у задачи должны быть критерии приёмки и артефакт проверки.
4. Записать задачи в `docs/execution/CURRENT_STAGE_TASKS.md`.
5. Выполнять задачи **по одной**:
   - взять задачу в `in_progress`;
   - реализовать;
   - прогнать проверки;
   - отметить задачу выполненной в `CURRENT_STAGE_TASKS.md`;
   - зафиксировать прогресс в `PROGRESS.md`.
6. Когда все задачи стадии закрыты, отметить стадию завершённой:
   - обновить статус в `PROGRESS.md`;
   - зафиксировать переход к следующей стадии.

## 2) Протокол работы с блокерами (обязательно)

Если возник блокер, агент должен:
1. Явно записать блокер в `docs/execution/PROGRESS.md`.
2. Указать **конкретные действия для снятия блока**:
   - кто должен сделать (agent / владелец проекта / DevOps);
   - что именно сделать (команда/настройка/роль/секрет);
   - критерий, по которому понятно, что блок снят.
3. Предложить минимальный обходной путь (если он безопасен и реалистичен).

Шаблон записи блокера:
- `Блокер`: краткое описание.
- `Владелец`: кто снимает блок.
- `Действия для снятия`: нумерованный список шагов.
- `Критерий снятия`: проверяемый результат (команда/лог/факт).

## 3) Что обязательно обновлять в документации

После каждой законченной задачи обновлять:
- `docs/execution/CURRENT_STAGE_TASKS.md` (статусы задач);
- `docs/execution/PROGRESS.md` (что сделано, что дальше);
- при необходимости `docs/PLAN.md` и `docs/STATUS.md`.

Если меняется процесс разработки/деплоя, обновить `README.md` с ссылками на актуальные документы.

## 4) Инженерные правила качества


- Перед коммитом запускать релевантные проверки (минимум `pytest -q`, если применимо).
- Не коммитить секреты, токены, реальные `.env`.
- Для инфраструктурных скриптов использовать `set -euo pipefail`.
- Для секретных шагов в CI запрещать вывод секретов в лог (`set +x`, mask).

## 5) Правила безопасной работы с YC/Lockbox

- Секреты берём только из Lockbox/GitHub Secrets, а не из git.
- Проверять минимум прав (least privilege) для service accounts.
- Любой деплойный скрипт должен иметь dry-run режим (где возможно).
- Все операции discovery/deploy документировать в `PROGRESS.md`.

## 6) Definition of Done для задачи

Задача считается выполненной только если:
1. Выполнены критерии приёмки из `CURRENT_STAGE_TASKS.md`.
2. Есть артефакт проверки (тест/команда/файл/лог).
3. Обновлён прогресс в `PROGRESS.md`.
4. Изменения закоммичены и оформлены PR.

## 7) Приоритеты по умолчанию

1. Снять блокеры, мешающие текущей стадии.
2. Завершить текущую стадию до начала следующей.
3. Сохранять прозрачность статуса через `docs/execution/*`.
4. Минимизировать риск утечки секретов и ручных операций.
